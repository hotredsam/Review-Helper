### Stale GitHub sync preview survives project switch and can apply project A's destructive actions to project B's repo
`src/components/SyncPanel.tsx:129` — found by react-state

MainPane renders panes without key={active.id} (MainPane.tsx:102-103), so when the user switches projects while on the Plan pane, the same PlanPane/SyncPanel instances are reused and SyncPanel's local `preview` state (set for project A) is never reset — there is no useEffect on project.id and no remount. If both projects' plans are already cached in planStore (PlanPane renders the full plan branch immediately, keeping SyncPanel mounted), the confirm button then calls syncMainApply(B.id, previewOfA). The Rust side replays the passed preview verbatim (sync/mod.rs:297 iterates preview.issue_actions; mod.rs:354-365 deletes exactly preview.file_deletions by path), so A's issue closes and .planning/* file deletions are executed against B's repo — deleting live files and closing wrong issues with no valid preview. This breaks the app's core guarantee that nothing destructive happens on GitHub without an explicit confirmed preview. Same missing-key root cause also leaks ChatPane's draft, PlanPane's kickoff description, and SyncPanel's success/error messages across projects.

**Evidence:** SyncPanel.tsx: `const id = project.id; const [preview, setPreview] = useState<SyncPreview | null>(null); ... onClick={() => void run("apply", async () => { const r = await syncMainApply(id, preview); ...` — preview is never invalidated when `project` changes; MainPane.tsx renders `<PlanPane project={active} />` with no key; src-tauri/src/sync/mod.rs: `for path in &preview.file_deletions { ... delete_file(&owner, &repo, path, ...) }` replays the stale preview against the new project's repo.

**Verifier:** Verified every link in the chain. SyncPanel.tsx:13-14,129 holds `preview` in local useState with no reset on project change and calls syncMainApply(project.id, preview) — current project id, possibly stale preview. MainPane.tsx:103 renders <PlanPane project={active} /> with no key, PlanPane.tsx:294 renders <SyncPanel project={project} /> with no key, and ProjectSwitcher/projectStore.select() only sets activeProjectId (section unchanged, no store reset) — so on a project switch while on the Plan pane, React reuses the same SyncPanel instance and A's preview survives, provided B's plan is already cached in planStore (otherwise PlanPane's `plan === undefined` loading branch unmounts SyncPanel; the finding correctly states this precondition). On the Rust side, sync/commands.rs:39 and sync/mod.rs:270-365 replay the client-supplied preview verbatim against the repo resolved from project_id: A's Close/Update issue numbers are applied to B's issues, A's Create actions create A's phase issues in B, and preview.file_deletions are deleted by path from B's default branch. SyncPreview carries no project id and there is no recompute/validation — no guard exists anywhere. This breaks the explicitly documented invariant (SyncPanel doc comment, commands.rs comment, CLAUDE.md: 'GitHub deletions/closes happen only after an explicit confirmed preview'). Severity adjusted from critical to major: file deletions are confined to `.planning/phases/*` path collisions (preview only lists that dir, mod.rs:254), so 'deleting live files' is overstated; all damage is recoverable via git history / reopening issues; and the flow requires both projects GitHub-linked, both plans cached in-session, and the user confirming a panel that visibly lists project A's items while B is active. Real, destructive, guard-defeating — but recoverable, scoped, and multi-precondition.

### Learning and Cards commands run multi-minute model subprocess calls synchronously on the main thread, freezing the whole app
`src-tauri/src/learning/commands.rs:111` — found by rust-backend

In Tauri 2, a non-async #[tauri::command] executes on the main thread (only `async fn` or #[tauri::command(async)] commands run off it; the repo has zero async commands — verified by grep). learning_intake, learning_propose, learning_notes, learning_flashcards, learning_quiz, and learning_tutor_send all block inside the command on a spawned `claude` CLI call that takes tens of seconds to minutes; cards do the same (card_explain at cards/commands.rs:95, card_clean_term, card_premade_questions, card_chat_send at :211). While any of these runs, the macOS event loop is blocked: every pane beachballs, no other invoke resolves, and the window stops repainting. The code comments only address the DB lock ("the model call runs WITHOUT the lock (so it never blocks the rest of the app)") — the main-thread freeze was missed. Grill/Plan/Chat got this right with spawn_guarded + events; the A–H Learning overhaul did not. Fix: make these commands async or move the model call to a spawned thread with events, like run_grill/run_chat.

**Evidence:** learning/commands.rs:84-115 — `#[tauri::command]\npub fn learning_intake(...) -> Result<Vec<IntakeItem>, String> { ... let questions = intake::fetch_questions(&subject)?; // model call, no DB lock held` (plain sync fn; same pattern at lines 155, 212, 239, 266, 363). cards/commands.rs:95 — `let content = generate_card(&term)?;` inside sync `card_explain`.

**Verifier:** Confirmed against the code. (1) Cargo.toml pins tauri = "2" and a repo-wide grep finds zero `async fn` commands and zero `#[tauri::command(async)]`; in Tauri 2 non-async commands run on the main thread. (2) learning/commands.rs: learning_intake (model call at line 111), learning_propose (155), learning_notes (212), learning_flashcards (239), learning_quiz (266), learning_tutor_send (363) are plain sync commands calling the model inline; cards/commands.rs: card_explain (generate_card at line 95), card_clean_term (154), card_premade_questions (168), card_chat_send (211) likewise. All are registered in lib.rs's invoke handler, so the freeze is reachable on first open of any of these views. (3) The call chain truly blocks: learning/gen.rs run_once → ClaudeCodeProvider::run spawns the `claude` CLI and synchronously reads stdout to completion with no timeout. (4) No guard exists elsewhere: LearningGate/CardGate are blocking mutexes that only serialize generation — they don't move work off-thread, and a concurrent caller would block the main thread on the gate too. gen.rs's own comment ("a blocking call here is safe and shows a spinner in the UI") shows the main-thread consequence was missed; the code comments only address the DB lock. (5) The contrast is accurate: grill/plan/chat/assess use spawn_guarded + events and return immediately. Minor overstatement only in "the window stops repainting" (WKWebView paints in a separate process, so CSS animations may continue), but the app's event loop is blocked: no other invoke resolves, window interaction freezes, macOS beachballs. Major (not critical) is the right severity: whole-app freeze for tens of seconds to minutes on routine Learning/Cards actions, but no data loss, corruption, or security impact, and the app recovers when the subprocess exits.

### sync_push_planning and sync_main_preview hold the global DB mutex across all GitHub network I/O — the one thing apply_main_sync was explicitly fixed to avoid
`src-tauri/src/sync/commands.rs:21` — found by rust-backend

There is a single app-wide `Db(Mutex<Connection>)` (db.rs:7) that every command and every background grill/plan/chat thread must lock. sync_push_planning locks it and then calls push_planning_branch(&conn, ...), which performs one branch_head_sha + ensure_branch plus a file_sha + put_file round-trip PER package file (sync/mod.rs:160-188) — with a 30s reqwest timeout each, a plan with 10 phases can hold the mutex for minutes on a slow connection. sync_main_preview does the same across list_issues + list_dir (sync/mod.rs:244-259). During that window every other command (project list, plan view, chat persistence in run_chat) blocks on the mutex, and since these are also sync commands the main thread is frozen too. apply_main_sync was deliberately restructured to take `&Db` and lock only briefly — its comment says "never across the slow GitHub network I/O in between, so a sync (or a hung request) can't freeze the rest of the app" (sync/mod.rs:266-269) — but the overhaul left the preview and planning-branch paths with the old pattern.

**Evidence:** sync/commands.rs:20-23 — `pub fn sync_push_planning(db: State<'_, Db>, project_id: i64) -> Result<usize, String> {\n    let conn = db.0.lock().map_err(|e| e.to_string())?;\n    push_planning_branch(&conn, project_id)\n}` and :27-30 same for `preview_main_sync(&conn, project_id)`; contrast sync/mod.rs:270-277 where apply takes `db: &crate::db::Db` and drops the lock before network calls.

**Verifier:** Verified all elements of the claim against the code. (1) The cited code exists verbatim: sync/commands.rs:20-23 and :27-30 lock the global Db mutex and pass the held guard into push_planning_branch / preview_main_sync. (2) Those callees perform sequential GitHub network I/O under the lock: push_planning_branch (sync/mod.rs:178-188) does branch_head_sha + ensure_branch plus a file_sha + put_file round trip per package file via push_files (mod.rs:160-174) — N+2 files for an N-phase plan; preview_main_sync (mod.rs:244-259) does list_issues + list_dir. github/mod.rs:23-24 confirms the claimed 10s connect / 30s request timeouts, so a slow connection can hold the mutex for minutes. (3) No guard exists: no spawn, no async, the MutexGuard lives for the entire command; SQLite busy_timeout does not apply to the Rust Mutex. (4) Contention is real: Db(Mutex<Connection>) at db.rs:7 has 105 lock sites across 17 files, including background threads spawned by chat/grill/plan (e.g. chat/commands.rs:160 persistence path via spawn_guarded), which would stall mid-stream while the lock is held. (5) The contrast claim is accurate: apply_main_sync (mod.rs:270-278, 337-340) was deliberately restructured to take &Db and lock only briefly, with the exact comment quoted in the finding — the codebase itself treats this pattern as a bug it already fixed once, and the preview/planning-push paths were left behind. Severity: major is appropriate — multi-minute app freeze plus stalled background workers on slow networks, but bounded by per-request timeouts and no data loss/corruption, so not critical.
